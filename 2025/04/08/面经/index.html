Java面向对象语言的三大特性

多态，封装，继承

插一嘴，如果问必须取消一个，可以取消哪一个？

应该是继承，原因：

  • 继承容易被滥用，导致设计僵化：子类强依赖父类，耦合度高，改动父类所有子类都会受到影响
  • 面向对象的设计，推荐使用组合（has xxx） 而不是继承（is xxx）

线程池的主要参数

  • corePoolSize:线程池的核心线程数量，默认情况下，如果线程池中的线程数量小于这个值，即使线程空闲也不会被摧毁
  • maximumPoolSize: 线程池最多可容纳核心数量。就是当阻塞队列满了，但是线程数量小于最大线程数，就会创建新线程。如果等于这个数量，就会执行拒绝逻辑
  • keepAliveTime: 保持线程存活的最大时间。当目前线程池中的线程大于corePoolSize，空闲时间超过这个值之后就会被销毁
  • unit: keepAliveTime的单位
  • workQueue: 工作队列，当没有空闲线程执行新任务的时候，放入的工作队列（会专门写一个介绍各种工作队列的）
    —————————————————— 以上是必须有的参数，下面是可选的 —————————————
  • threadFactory：线程工厂，可以用来给线程取名
  • handler: 拒绝策略。线程执行拒绝的方法（下面有介绍拒绝策略）

线程池处理任务的流程

线程拒绝策略

  • AbortPolicy: 直接抛出RejectedExecutionException异常，如果不指定，默认的异常就是这个
  • CallerRunsPolicy: 线程调用者所在的线程去执行被拒绝的任务（相当于直接拿当前接口的线程去执行这个任务，可能会卡住主进程）
  • DiscardPolicy：直接丢弃
  • DiscardOldestPolicy：丢弃最老的任务，然后执行该任务
  • 继承RejectedExecutionHandler接口：自定义任务的拒绝策略

怎么设置线程池大小

cpu密集型：corePoolSize = CPU核数 + 1

IO密集型：corePoolSize = CPU核数 * 2

线程池的种类

一下都是Executors提供的线程池工厂方法，但是阿里开发手册中是不建议这种直接

ScheduledThreadPool: 设置定期执行的任务，支持定时或周期性执行任务

FixedThreadPool: 固定线程数量的线程池，线程数量是固定的（相当于corePoolSize == maximumPoolSize），不会扩容和销毁

CachedThreadPool: 缓存线程池，线程数可以无限增大，所以容易OOM

SingleThreadExecutor: 使用唯一的线程去执行任务，可以保证任务执行的顺序跟提交的顺序一样

synchronized工作原理

是java的一个关键字，底层是由JVM实现的，依赖于Monitor（监视器对象），在编译之后的同步代码块的前后加上Monitor enter 跟 Monitor exit字节码指令，由于依赖于操作系统底层互斥锁实现，所以是排他锁，会导致线程阻塞从用户态向内核态的转变，所以性能消耗很大

Reentrantlock工作原理

java.util.concurrent.locks.ReentrantLock

底层依赖于AQS（AbstractQueuedSynchronizer）的, 这个抽象类提供了同步的基本框架，包括了队列，状态值等

  • 可中断性：线程在等待锁的过程中，可以被其他线程中断而提前结束等待
  • 设置超时时间：支持在尝试获取锁时设置超时时间
  • 公平锁和非公平锁：默认情况下是非公平锁（多个线程竞争），但是可以传入参数true设置为公平锁（按照线程来的先后次序获取锁跟释放锁）
  • 可重入性：同一个线程可以多次获取同一把锁，而不会造成死锁
    （这里看小林coding的有点问题，有getHoldCount方法可以拿到当前线程拿锁的数量，但是并不是用的ReentrantLock里面的有一个变量HoldCount，而是AQS中，定义了一个state变量，可以理解成锁的计数器，如果 == 0，表示锁是空闲的）

synchronized和reentrantlock区别？

  • 用法不同：一个可以用来修饰方法和代码块，一个只能在代码块中使用
  • 获取锁的方式不同：前者会自动加锁释放锁，后者只能手动加锁释放锁
  • 锁类型不同：一个只能非公平锁，一个可以公平跟非公平
  • 响应中断不同：后者可以响应中断，解决死锁问题
  • 底层实现不同：前者是使用的jvm层面的Monitor实现，后者是AQS

浏览器输入网址到呈现页面的过程

7个步骤

  1. 浏览器解析url，解析是什么
    1. 解析开头的协议（http / https）
    2. 解析端口/路径/参数
    3. 根据url解析的信息，生成http请求消息
  2. DNS域名解析
    1. 先查本地DNS缓存，如果本地缓存中有ip，就直接返回ip地址
    2. 本地没有，就递归的查询根域DNS服务器（.），顶级域DNS服务器（.com），权威域DNS服务器（server.com）。最终得到目标服务器的ip
    3. 浏览器， 操作系统都会有缓存，先问本地缓存，再问本地DNS服务器
  3. TCP三次握手建立连接（http是TLS + TCP）
    1. 客户端 —-> 发送SYN包 —-> 服务器
    2. 客户端 <—- SYN + ACK 包 <—- 服务器
    3. 客户端 —-> ACK 包 —-> 服务器
  4. 如果是https，还会与服务器进行TLS / SSL 握手：
    1. 交换证书
    2. 生成对称加密密钥
    3. 验证服务器是否合法（数字证书）
    4. 建立加密链接
  5. 浏览器发送http请求
  6. 服务器响应http请求（查找资源，处理逻辑），并返回响应
  7. 浏览器渲染页面
